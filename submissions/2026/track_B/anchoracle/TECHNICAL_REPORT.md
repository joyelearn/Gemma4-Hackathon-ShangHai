## 1. 项目目标与痛点

**痛点**：游客在景点面对一座雕像或一栋老建筑时，通常只能看见一块字数有限、文绉绉的介绍牌。无法对话、无法深入、无法跟随这片土地真正的"人"去理解它。

**解决方案**：让真实的历史人物本人作为 AI 角色，**亲自向你讲述他生活过的城市**。  
苏轼讲西湖，鲁迅讲鲁迅公园 —— 由 Gemma 4 完成从"识别 → 锁定人物 → 角色扮演"的全链路。

---

## 2. 模型分工：两个 Gemma 4 变体各司其职

视觉识别与历史人物对话是两类诉求截然不同的负载：前者要"一次看准"，后者要"持续、低延迟地表达"。我们用 Gemma 4 家族中的两个变体分别承担，让每条链路都跑在最契合它的模型上。

### 2.1 视觉感知：Gemma 4 31B IT (Dense)

视觉识别是整个产品最重的一环——一张照片进来，要在有限时间内同时读出文字、判断场景、识别主体，并从 30 位历史人物中认出雕像是谁。这一环由 **31B Dense** 变体承担：

- **用 Dense 而非 26B A4B（MoE）**：识别成败常取决于细节——碑文小字、雕像面部特征、牌匾字形。Dense 全参数激活在这类细粒度多模态判别上的一致性，明显优于稀疏激活的 MoE；而景点 / 雕像识别"差之毫厘谬以千里"，稳定性比单次延迟更重要。
- **256K 长上下文撑起"单次推理多任务"**：我们把完整人物清单（30 人）、景点清单（90 个）、场景类型枚举一次性写进同一个 prompt，让 Gemma 4 31B 在**一次推理**中并行产出 OCR、场景分类、主体识别、30 选 1 人物识别四组结构化结果。传统做法要串接四个模型、逐级解析，这里压成一次调用：既省去多轮往返延迟，也让四项判断基于同一份视觉理解，避免级联误差。

> 偶发不可达（罕见 timeout / region 故障）时自动降级到 `gemini-2.5-flash` 兜底，保证识别链路始终可用。

### 2.2 历史人物对话：Gemma 4 26B A4B IT (MoE)

进入对话后，诉求变成"30 位人物、各自的口吻、还要逐字流式吐出"。这一环换用 **26B A4B 的 MoE** 变体：

- **MoE 的低延迟天然适配流式对话**：每次推理仅激活约 4B 参数，首字延迟低、吞吐高；配合 SSE 逐字渲染，用户几乎即时看到人物"开口"，而非干等一整段生成完毕。
- **26B 总量托住人物风格**：苏轼的文言、鲁迅的民国白话、秦良玉的西南语感——要让 30 个人各说各话且不出戏，需要足够的参数量稳住语言风格；26B 的总容量恰好在"够稳"与"够快"之间取得平衡。

### 2.3 小结

**重判别的视觉交给 31B Dense，重表达的对话交给 26B A4B MoE。** 同一个 Gemma 4 家族的两种形态，正好对应产品的两类核心负载，也是整套链路能既"看得准"又"答得快"的根本。

## 3. 架构分层与模块边界

项目按职责分为四层，业务编排集中于 API 层，领域数据与基础设施彼此独立、互不耦合：

```
┌─ 表现层 (src/app/*/page.tsx)
│    scan 拍照页 · chat 对话页 · city 城市页 · 首页
│    仅负责渲染、交互与 SSE 消费，不承载业务规则
├─ API 编排层 (src/app/api/*/route.ts)
│    scan/route.ts  —— 视觉推理 + 多信号加权评分
│    chat/route.ts  —— 流式对话 + 实用查询挂钩
├─ 领域数据层 (src/data/*.ts)
│    characters / landmarks / landmarkAliases / personas —— 30 人物 · 90 景点 · 别名 · 人设
│    纯数据与类型枚举，识别与对话逻辑均 import 此层
└─ 基础设施层 (src/lib/*.ts)
     aiClient(Gemma/Gemini 单例) · chatDb(IndexedDB) · imageUtils(压缩)
```

在文件粒度上，我们做过一次审慎权衡。`scan/route.ts` 行数偏多，我们评估过将其拆分为 scoring（评分）、vision（视觉调用）、prompts（提示模板）等多个文件；但它内部已由一组职责单一、低耦合的纯函数构成（`parseVisionJson`、`subjectTypeBonus`、`withTimeout` 等），属于"文件内高内聚"而非逻辑缠绕。考虑到这条链路已经过反复验证，赛期我们将其稳定性置于行数规范之上，选择保留现有编排——这是一次在工程规范与交付稳定之间的有意识取舍。

数据流清晰单向：照片进入 `/api/scan`，Gemma 31B 单次推理叠加加权评分锁定景点，再跳转至 `/api/chat`，由 Gemma 26B 以人物口吻流式作答。新增城市或人物时，只需扩展领域数据层，编排层与基础设施层无需改动——这正是我们分层设计的初衷。

## 4. 多模态识别架构

### 4.1 单次推理完成四个子任务

传统做法是串行调用四个模型（OCR → 分类 → 雕像识别 → 检索）。本项目利用 Gemma 4 31B 的 256K 上下文，将所有任务**编码进一次 prompt**：

```
prompt = scene_type_enum + subject_type_enum + character_closed_list(30人) + visual_hints
image = user_upload
↓
Gemma 4 31B 单次推理输出 JSON：
{
  "text_on_image": ["1881-1936"],          // OCR
  "scene_type": "纪念广场",                  // 10 类场景分类
  "subject_type": "人物雕像",                // 6 类主体分类
  "person_in_statue": "鲁迅",                // 30 选 1 人物识别
  "notable_features": ["蓄短发八字胡", "圆框眼镜"],
  "possible_name": "鲁迅公园"
}
```

**收益**：
- API 调用次数：4 → 1（成本与延迟均降低 ~75%）
- 任务一致性：所有判断基于同一视觉理解，避免串行模型的累积误差

### 4.2 多信号加权评分（解决"故居 vs 公园"难题）

**问题**：鲁迅故居和鲁迅公园都属于鲁迅，单靠人名识别会混淆。

**早期失败方案**：级联 first-match 规则（OCR 命中 → 人物匹配 → 场景兜底）。结果：上海鲁迅公园被识别成鲁迅故居（因为人物相同时优先级混乱）。

**当前方案**：对每个候选景点计算综合得分：

| 信号 | 分值 | 说明 |
|------|------|------|
| OCR 景点名直接命中 | **+10** | 例如照片有"鲁迅公园"四个字 |
| 别名命中（如"西子湖" → 西湖） | +3 ~ +6 | 按渠道权重 |
| Gemma 识别出雕像人物 | **+12** | 该人物拥有的所有景点全部加分 |
| `possible_name` 命中 | +3 | Gemma 给出的猜测 |
| `notable_features` 命中 | +2 | 视觉特征 |
| 场景类型匹配 | **+4** / **−2** | 鲁迅公园声明为 `["开阔公园", "园林"]` |
| 主体 × 景点类型冲突 | **−8 (决定性)** | "人物雕像" × "故居" → 否决（故居室内不会有大型雕像） |

**冲突检测**：`subjectTypeBonus()` 函数将"主体类型"与"景点类型"做语义冲突检测，例如：
- `subject_type = "人物雕像"` + `landmark.type = "故居"` → **−8 决定性否决**
- `subject_type = "人物雕像"` + `landmark.type = "公园"` → +4
- `subject_type = "建筑外观"` + `landmark.type = "故居"` → +4

这一步彻底解决了"故居 vs 公园"混淆问题。

### 4.3 数据模型扩展

为了支持 4.2 的评分，我们给每个景点增加两个语义字段：

```typescript
export interface Landmark {
  name: string;
  description: string;
  type: LandmarkType;      // 11 类：故居 / 纪念馆 / 公园 / 陵园 / 广场 / 园林 / 古镇 / 校园 / 历史建筑 / 自然景观 / 城市地标
  sceneTypes: SceneType[]; // 10 类：室内陈列 / 民居院落 / 开阔公园 / 纪念广场 / 园林 / 自然景观 / 城市街景 / 历史建筑 / 校园 / 古镇街区
}
```

`type` 和 `sceneTypes` 的双重声明，让一个"人物雕像"主体能精准对应到该人物的**公园**而非**故居**。

## 5. Gemma 4 的运用，以及为何不采用 function calling

我们对 Gemma 4 的运用并非简单的单轮问答，而是围绕其长上下文与原生多模态能力做的针对性设计。

一张照片进入后，Gemma 31B 在**单次推理**中同时完成 OCR、场景分类、主体判断，以及"从 30 位历史人物中识别出雕像是谁"。传统方案需串行调用四个模型，而我们借助其长上下文与多模态能力，将其合并为一次调用。人物识别被刻意约束在 30 人的闭卷清单内，以显式收窄模型的输出空间，从而有效抑制幻觉。

而"最终锁定哪个景点"这一决策，我们没有交给模型，而是落在代码侧，由多信号加权评分与"主体 × 景点类型"冲突检测共同得出。这样做的考量很明确：评分中的每一分都可追溯、可复现，一旦结果异常，也能依据日志定位偏差出现在哪一环。我们希望把关键决策保留在可控、可解释的代码层。

至于为何不采用 function calling，我们经过认真评估后选择不用——并非能力所限，而是基于三点判断：

- **任务契合度**：本场景的链路是"识别 → 评分 → 锁定"，决策确定且要求可复现，更适合实现于代码侧。function calling 的价值在于让模型动态选择调用何种工具，而此处动作固定，交由模型反而削弱了可控性。
- **稳定性**：Gemma 4 在 AI Studio 上的原生函数调用支持尚未经我们充分验证。赛期将已跑通的核心链路推倒重构、转而依赖一项不确定的能力，并非稳妥之选。
- **按需使用工具**：在确实适合的环节，我们同样使用了工具——对话中涉及交通、门票等实时信息时，即通过 Gemini 的 googleSearch 完成检索。因此这并非不会用工具，而是依场景取舍。

这套"约束式结构化生成 + 确定性评分层 + 按需检索"的组合，换来的是一条可复现、可解释、演示稳定的多模态管线。对一个面向真实用户的产品而言，这样的工程判断，比强行引入 function calling 更有价值。

## 6. 对话子系统

### 6.1 角色扮演

每位历史人物有独立的 `Persona`：
- 朝代 / 称谓 / 生平
- 语言风格（古汉语 / 民国白话 / 当地方言）
- 守护景点清单
- 全局约束（不出戏 / 不混用现代术语 / 不离开自己的时代）

### 6.2 流式输出

使用 SSE（Server-Sent Events）将 Gemma 4 26B A4B 的输出分片返回，前端逐字渲染。

### 6.3 实用查询挂钩

对话中如检测到关键词（交通 / 门票 / 美食 / 住宿），后端会附加高德地图 deep-link，让历史人物的话术与现实工具自然衔接：

> 鲁迅："实事求是，不妨直接查看 → 高德地图·查看路线"

### 6.4 语音输入

Chat 页集成 Web Speech API，支持中文语音转文字，凑齐"视觉 + 文本 + 音频"三模态。

## 7. 工程亮点

| 维度 | 实现 |
|------|------|
| **状态管理** | React 19 + URL state（无重型状态库） |
| **聊天记录** | IndexedDB 本地持久化（隐私优先） |
| **图片上传** | 客户端压缩（5MB 上限）→ 减少 base64 传输 |
| **错误处理** | 配额识别 / timeout 容错 / SSE 解析降级 |
| **可观测性** | 服务端日志带 `[Scan]` 前缀，便于 Vercel 日志过滤 |
| **安全** | API key 仅服务端读取，前端零暴露；CSP 默认严格 |
